# System Architecture

## Purpose

This project is a local AI workflow runner for controlled development and review tasks. It provides a ChatGPT-like web UI, workflow designer, workflow asset library, CLI entrypoint, and a Python-controlled runtime that can call agent providers such as Qwen CLI or OpenCode while keeping verification, retry, artifacts, and workspace safety under deterministic Python control.

The main design principle is:

```text
AI agent = understand, plan, implement, explain failures
Python runner = orchestrate, isolate, validate, retry, gate, report
Human = final review / accept changes
```

## High-Level Components

```text
Browser UI
  ├─ Chat / sessions / workflow runner
  ├─ Workflow Designer
  └─ AI Workflow Assets editor
        ↓ HTTP / SSE
FastAPI app
  ├─ API routes
  ├─ Services
  ├─ Runtime modules
  ├─ Workflow runtime
  ├─ Agent providers
  ├─ Python validation functions
  └─ JSON store / workspace artifacts
        ↓
Selected Project Path
  ├─ production files
  ├─ generated tests
  ├─ optional validation script
  └─ architecture.md generated by workflow
```

## Workflow Asset Model

Workflow behavior is data-driven under `data/ai-workflow/`:

```text
data/ai-workflow/
  workflows/
    *.workflow                    # workflow step list
  contracts/
    <workflow>/<step>.yaml         # step type, agent, retry, output, functions, review/gate settings
  steps/
    <workflow>/*.md                # prompt templates
  functions/
    common/*.py                    # reusable Python function assets
```

The runtime loads workflows through `workflow_asset_service`, normalizes step configuration, then `WorkflowExecutor` executes steps in order.

## Runtime Execution Flow

```text
Start workflow run
  ↓
Load workflow asset and contracts
  ↓
Create run workspace
  ├─ requirement.md
  ├─ input/
  ├─ output/
  ├─ prompts/
  └─ .workflow/state.json
  ↓
For each enabled step
  ├─ Build prompt from requirement, artifacts, project index, failure feedback
  ├─ Run AI / Python / gate / review action
  ├─ Validate expected files and project write constraints
  ├─ On failure, classify retry target
  └─ Append structured failure feedback and retry from owner step
  ↓
Write final state, trace artifacts, and UI events
```

## General Auto Development Workflow

Current step order:

```text
prepare_project
  ↓
plan_tasks
  ↓
implementation_review
  ↓
build
  ↓
generate_tests
  ↓
run_test
  ↓
run_external_validation
  ↓
final_review
  ↓
final_gate
```

### Current responsibilities

| Step | Type | Responsibility |
| --- | --- | --- |
| `prepare_project` | AI + deterministic fallback | Reads project context, generates/updates `architecture.md`, and writes Python-generated `output/project-index.md`. |
| `plan_tasks` | AI | Splits the requirement into small tasks with acceptance criteria, dependencies, assembly plan, and stop conditions. |
| `implementation_review` | Python | Validates/repairs `todo.md`, generates `output/task-manifest.md`, and gates whether Build may proceed. |
| `build` | AI | Produces production/project `FILE/CONTENT/END_FILE` blocks only. Test files are rejected in this step. |
| `generate_tests` | AI + generic fallback | Produces focused generated tests under the test folder only. Production files are rejected in this step. |
| `run_test` | Python | Runs the configured/default test command and writes `output/test-result.md`. |
| `run_external_validation` | Python | Runs run-specific or fallback validation scripts and writes `output/external-validation-result.md`. |
| `final_review` | Python | Verifies concrete artifacts, tests, validation, and task manifest instead of trusting AI self-review. |
| `final_gate` | Gate | Requires `final-review.md` to contain `Status: PASS`. |

## Small-Task Development Model

`general-auto-development` now treats task splitting as a deterministic artifact chain:

```text
AI todo.md
  ↓ Python implementation_review
output/task-manifest.md
  ↓
Build prompt receives task order + assembly strategy
  ↓
Generate Tests prompt receives task order + assembled behavior
  ↓
Final Review checks task-manifest + test result + validation result + build artifact
```

This is intentionally lighter than a fully dynamic per-task loop. It gives the agent a stable small-task order while preserving the existing simple workflow shape.

## Verification Model

Final success is not based on AI saying OK.

Required evidence:

```text
output/task-manifest.md            # Python-generated task order exists
output/build-result.md             # Build returned production FILE blocks
output/test-result.md              # Automated test command passed
output/external-validation-result.md# External validation passed or skipped by config
output/final-review.md             # Python deterministic review says Status: PASS
```

Validation sources:

| Gate | Owner | Pass condition |
| --- | --- | --- |
| Path guard | Python | Generated writes stay inside selected Project Path. |
| Build file ownership | Python | Build writes production/project files only, not tests. |
| Test file ownership | Python | Generate Tests writes test files only. |
| Automated tests | Python command runner | Test command exits with code 0. |
| External validation | Python command runner | Validation script exits with code 0 or is explicitly skipped. |
| Final review | Python function | Concrete artifacts show PASS evidence. |

## Retry / Repair Model

Failures are routed back to the owner step instead of blindly restarting the workflow.

Examples:

```text
Build output missing production FILE blocks  → retry build
Generated tests have import/syntax issues    → retry generate_tests
Tests fail due behavior mismatch             → retry build
External validation fails                    → retry build
Final review missing evidence                → retry build or prior owner step
```

Structured feedback is written to:

```text
input/failure-feedback.md
```

Repeated identical failures are still allowed to retry until the configured retry limit because small/local models may recover on later attempts. Workspace isolation failures remain hard safety failures through path guards.

## Workspace Isolation

Selected Project Path is the only write root.

Rules:

```text
Allowed:
- Read project files.
- Read external reference files when needed.
- Write generated production/test/artifact files under Project Path only.

Rejected:
- Absolute FILE paths.
- Parent traversal `..`.
- Writes into `.git`.
- Writes into `.ai-workflow` and `.qwen-workflow`.
- Build writing tests.
- Generate Tests writing production files.
- Build overwriting existing validation scripts unless explicitly requested.
```

## Run Profiles

Run profiles tune runner behavior without adding extra workflow steps.

| Profile | Intended use | Behavior |
| --- | --- | --- |
| `fast` | quick iteration | Fewer retries, thinking disabled. |
| `normal` | default | Uses workflow default retry and step settings. |
| `deep` | small-model or harder task | Enables compatible-agent thinking flags and raises retry floors for key steps. |

CLI example:

```bash
python -m app.cli.aiwf run "add config checker" --project . --workflow general-auto-development --profile deep
```

## Extension Points

| Need | Extension point |
| --- | --- |
| Add a new workflow | Add `.workflow`, contracts, and step prompts under `data/ai-workflow/`. |
| Add a reusable validation/check function | Register a function in `app/workflow_runtime/builtin_functions/registry.py` or add a workflow Python asset. |
| Add an agent provider | Implement an adapter under `app/workflow/agents/providers/` and register it. |
| Add UI workflow fields | Update Workflow Designer model/settings renderers and backend schema normalization. |
| Add stricter verification | Add Python gates in builtin functions or workflow-specific action logic. |

## Current Stability Assessment

Strengths:

- Workflow definitions are data-driven.
- AI implementation is separated from Python validation.
- Project Index and Task Manifest reduce model guessing.
- Build/test ownership checks prevent common output mixing.
- External validation can be user/project supplied.
- Workspace write isolation is enforced before materializing file blocks.
- Retry feedback is structured and routed to the failed owner step.

Remaining improvement areas:

- A fully dynamic per-task loop is not implemented yet; current task splitting stabilizes prompt context and final gates but Build still runs as one step.
- Sandbox/worktree staging before applying changes to the real project would further reduce risk.
- Diff summary and human apply gate can be made more explicit.
- Test command detection can be expanded for non-Python projects.


## V5 Coding-Agent Control Upgrade

`general-auto-development` now has a stronger controlled coding path without enabling sandbox/worktree mode:

```text
Prepare Project
→ Plan Tasks
→ Implementation Review / task-manifest.md
→ Build per-task loop
→ Generate Tests per-task loop
→ Run Test
→ External Validation
→ Evidence Verifier / verifier-report.json
→ Diff Reviewer Agent
→ Final Gate
```

Important constraints:

- The workflow does not run `git commit` or `git push`; users inspect git diff themselves.
- Build and Generate Tests materialize task-level artifacts under `output/tasks/<TASK-ID>/`.
- `output/verifier-report.json` is the deterministic pass/fail evidence.
- `diff_review` is advisory only; it can warn about risks but cannot override the verifier.
- Retry feedback is classified so small/local models can repair with a new strategy without stopping too early.
